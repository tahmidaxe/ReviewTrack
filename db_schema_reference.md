# Reviewer Dashboard Database Schema Reference

This document provides a simple explanation of the SQLite database generated by the Background Updater, to be used by the PyQt6 frontend interface holding the review dashboard.

## Database Location
- **Device location:** `~/.local/share/BackgroundFetcher/reviews.sqlite3`
- The database is fully accessible over the network (e.g. over SSH, Samba, or depending on how your directory is shared).
- It runs with `PRAGMA journal_mode=WAL` (Write-Ahead-Log). This means the UI frontend app can read from the DB concurrently while the background network fetcher safely writes to it natively, preventing lockups without any extra IPC layers.

---

## Core Table: `review_items`
This is the single table you need to query to populate the dashboard UI. The table retains a continuous record of academic review requests over their different lifecycle stages.

### 1. Parsed Review Invite Information 
*(When the researcher first receives an invitation by a journal to review)*
* **`status`**: Will be set to `"invited"`
* **`journal_name`**: The name of the academic journal sending the request (e.g., "IEEE Access").
* **`paper_id`**: The unique identifier/manuscript number for the paper.
* **`paper_title`**: The full title of the submitted manuscript.
* **`paper_abstract`**: A brief summary of the paper if detected in the email body.
* **`date_invited`**: ISO 8601 timestamp string representing when the invitation email was originally received.
* **`review_due_date`**: ISO 8601 representation (YYYY-MM-DD) for when the journal is requesting the review back by.
* **`agree_link`**: Unique URL for the researcher to click to instantly *Accept* the review assignment.
* **`decline_link`**: Unique URL for the researcher to click to instantly *Decline* the review assignment.

### 2. Parsed Review Accepted Information
*(After the researcher accepts an invite, changing it to a 'Pending Task')*
* **`status`**: Will be set to `"accepted"` (marks the paper as actively being reviewed).
* **`journal_name`**: The name of the journal.
* **`paper_id`**: The unique identifier for the paper. 
* **`paper_title`**: The title of the paper.
* **`date_accepted`**: ISO 8601 timestamp string representing when the official acceptance confirmation or reminder email was received.
* **`manuscript_portal_link`**: The generic URL to log back into the reviewer center (e.g., Editorial Manager, Manuscript Central) to download resources and submit the final review.
* **`direct_review_link`**: Some journals provide a direct URL to bypass login to view the PDF or score sheet directly.

### 3. Parsed Review Completed Information
*(After the researcher submits their review to the journal)*
* **`status`**: Will be set to `"completed"` (signals the UI to move it to the historical view).
* **`journal_name`**: The name of the journal.
* **`paper_id`**: The corresponding paper ID of the finalized review.
* **`paper_title`**: The title of the finalized paper.
* **`date_completed`**: ISO 8601 timestamp string representing when the "Thank You" or confirmation email was received from the journal after submission.

### Basic Metadata Fields Reference
* **`id`**: Standard SQL Primary Key integer. Note: use `paper_id` for tracking uniquely between emails, rather than `id`.
* **`last_updated`**: The timestamp representing the exact moment the BackgroundFetcher LLM updated/upserted this row in the table.

## Usage Guide for Frontend App
When creating the layout for the 800x480 touch display:
1. **Invites Widget**: Select queries where `status = 'invited'`. Display the title + Accept / Decline URLs.
2. **Pending Tasks Widget**: Select queries where `status = 'accepted'`. Display the Due date, title, and Log in URL.
3. **Completed History Widget**: Select queries where `status = 'completed'`. Display chronological history based on `date_completed`.
